# Architecture Rationale

This recipe browser is intentionally structured around small, focused functional components with typed interfaces so each concern can evolve independently. The top-level composition starts in `main.tsx`, where `BrowserRouter` and the `FavoritesProvider` wrap the app. From there, `App.tsx` handles route definitions only, while `Layout` provides shared navigation and page chrome using the `children` composition pattern. Feature components are split by responsibility: `RecipeList` maps collections, `RecipeCard` renders summary items, `RecipeDetailPage` handles record-level rendering with route params, and `RecipeForm` encapsulates controlled form behavior.

State placement follows scope and reuse needs. Local, short-lived state (form values, validation errors, latest submission preview) lives inside the relevant component via `useState`. Cross-route state (favorite recipe IDs) is promoted to Context through `FavoritesContext` so it can be consumed by `Layout`, `RecipeCard`, and detail views without prop drilling. This keeps parent components simpler and avoids threading favorite handlers through unrelated layers.

Two custom hooks keep logic reusable and testable. `useRecipesData` encapsulates async data loading and the three-state pattern (`loading`, `error`, `recipes`) with `useEffect` cleanup via `AbortController` to prevent state updates after unmount. `useLocalStorage` centralizes persistence behavior and powers the favorites context, so storage concerns are not duplicated across UI components.

Routing is organized with React Router into three clear views: list (`/`), detail (`/recipes/:id`), and add (`/add`). The detail route uses `useParams` to select the active recipe from fetched data. Data is fetched from local JSON in `public/recipes.json` to keep the project self-contained while still demonstrating asynchronous flows and error handling. Testing targets meaningful behavior: rendering/navigation, validation-driven form outcomes, and interaction-based favorite toggling.
